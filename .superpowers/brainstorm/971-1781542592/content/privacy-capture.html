Privacidade & captura — controle pelo dashboard

Nova seção na edição do projeto. A infra (parent→child + precedência .env > dashboard > default) já existe; falta a UI + validação + testes. Mockup e decisões abaixo.

Privacidade & captura · projeto "API"
Capturar PIIe-mails, nomes e dados pessoais nos payloads
Capturar corpo de e-mailsconteúdo das mensagens enviadas (mail body)
Capturar PII ⚠ fixado pelo .env do child (true) — toggle ignorado
🔒 Piso de credenciais (senhas, tokens, CPF, API keys) é sempre mascarado. Desligar isso (disable_credential_scrub) grava segredos em texto puro — fica disponível apenas via .env do child, não no dashboard.

Decisões

1

Expor PII e mail_body como toggles

Os dois flags "normais" ficam controláveis pelo dashboard.

2

disable_credential_scrub NÃO no dashboard

Categoria perigosa (credencial ≠ PII). Fica só via .env, com aviso explicativo na UI. (recomendado)

3

Mostrar de verdade quando o .env está vencendo

O child reporta no ingest quais knobs estão fixados no .env; a UI marca "⚠ ignorado: definido no .env do child". Evita o toggle mentir silenciosamente. (recomendado, custa um pouco mais)

4

Alternativa barata: só aviso genérico

Sem reporte do child — apenas um texto fixo "se o .env do child definir, ele prevalece". Marque em vez de #3 se quiser o caminho mais simples.

5

Confirmação ao LIGAR captura sensível

Ligar "Capturar PII" pede confirmação (modal) — é uma ação com implicação de privacidade.

Recomendação: 1 + 2 + 3 + 5. Expõe PII/mail_body com confirmação ao ligar, mantém o piso de credenciais fora do alcance da UI, e mostra honestamente quando o .env do child está no comando. #4 só se você preferir não mexer no payload de ingest agora.
