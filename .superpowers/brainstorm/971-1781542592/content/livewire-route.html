Rota real por trás de livewire/update

Hoje toda interação Livewire vira a rota técnica livewire.update — você não sabe qual página o usuário estava. A página real está no header Referer (que o Warden não captura hoje). Três níveis de solução, combináveis.

A

Capturar a página de origem (Referer) barato

Gravar o Referer no payload da request. Mostra "veio de /orders/42". Base para tudo. Passa pelo Scrubber (URL pode ter PII na query string).

POST livewire.update  →  origem: /orders/42
B

Resolver para a ROTA real médio

No child, casar a URL do Referer com o router (Routes->match()) para obter o nome da rota de origem (orders.show) — não só a URL. Vira o label e pode reagrupar as métricas por página real.

livewire.update  →  orders.show (via Livewire)
C

Extrair o componente + ação Livewire mais frágil

Ler do corpo da POST o nome do componente e o método chamado (OrderList::delete). O mais preciso, mas depende do formato interno do snapshot Livewire (pode quebrar entre versões).

OrderList › delete()  @  orders.show

Como exibir o agrupamento?

1

Reagrupar por página real

Cada página Livewire vira sua própria linha em Top Routes (orders.show, products.index…). Você vê qual página gera mais carga/erro. Pode fragmentar se houver muitas páginas.

2

Manter livewire.update + anexar origem

A rota continua livewire.update nas métricas, mas cada trace/erro mostra a origem. Menos disruptivo, mas Top Routes continua "cego".

Recomendação: A + B, com agrupamento (1). Captura o Referer e resolve para a rota real, reagrupando as métricas por página — é o que responde "qual página o usuário estava". C fica como enriquecimento opcional (preciso mas frágil). Tudo respeitando privacidade: o Referer passa pelo Scrubber e a query string sensível é redigida conforme capture.pii.
