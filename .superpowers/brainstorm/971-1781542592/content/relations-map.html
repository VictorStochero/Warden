Quais saltos entre dados (drill-down) entram na 0.3.1?

O Trace (waterfall) é o hub natural — quase tudo deveria levar até ele e voltar. Abaixo, o mapa atual: verde = já existe, vermelho tracejado = falta. Selecione os saltos que faltam e que você quer na 0.3.1.

Top Routes ⇢ Requests da rota ⇢ TRACE (waterfall) → Span → Event bruto
Slow / Frequent Query ⇢ Traces onde rodou → TRACE
Error 5xx ⇢ TRACE ⇠ HTTP host / Job / Cache
Exception (span no trace) ⇢ Issue → Eventos da issue → Incident
Incident → Issues relacionadas ⇢ Trace do 1º erro
Deploy / Release marker ⇢ Erros & issues desse deploy
→ salto já existe  ·  ⇢ salto que falta

Saltos faltantes — quais priorizar? (marque vários)

1

Rota → seus traces/requests

Clicar numa linha de "Top Routes" mostra os traces recentes daquela rota. Hoje a tabela é um beco sem saída.

2

Query lenta → traces onde apareceu

Da query lenta para os traces reais que a executaram — fecha o "onde isso roda?".

3

Erro 5xx → trace

Cada erro recente abre direto o waterfall da request que falhou.

4

Exception no trace ↔ Issue

Span de exceção no waterfall linka para a Issue agrupada (e a Issue volta ao trace).

5

Incident → trace do 1º erro

Do incident para a evidência concreta (primeiro trace que disparou).

6

Deploy marker → erros/issues do deploy

Clicar no marcador de deploy filtra erros e issues surgidos naquela release.

7

HTTP host / Job / Cache → traces

Mesmo padrão da rota/query, para as demais tabelas de agregado.

Recomendação: 1, 2, 3, 4 são os de maior retorno — transformam tabelas "mortas" em pontos de partida de investigação e fecham o ciclo agregado → trace → issue. 5, 6, 7 são ótimos complementos se couber no escopo.
