Auditoria de queries — quais "problemas" detectar automaticamente?

Tudo abaixo é detectável zero-dep com o que o Warden já captura (SQL normalizado + bindings + trace_id + duração), via análise estática do SQL + agregação por trace. Marque os que entram na seção Database.

1

N+1 altodetector já existe

Mesma query (normalizada) repetida N× no mesmo trace. Já detectado por-trace; falta o relatório de projeto + listar traces afetados.

2

Queries idênticas duplicadas médio

Mesma SQL e mesmos bindings repetida no trace — não é N+1, é desperdício puro (cache de request perdido). Ex.: a mesma SELECT user 42 rodando 6×.

3

SELECT * (over-fetching) baixo

Query que puxa todas as colunas. Regex no SQL. Sinaliza hidratação desnecessária.

4

UPDATE/DELETE sem WHERE crítico

Escrita em massa sem cláusula WHERE — risco de apagar/alterar a tabela inteira. Regex no SQL. Raro, mas quando aparece é grave.

5

Request "gorda" — volume de queries médio

Trace com contagem total de queries acima de um limiar (ex.: >50 numa request) mesmo sem N+1 — indica página/endpoint pesado.

6

Queries lentas médiojá existe

Acima de slow_query_ms (default 100ms). Já temos; só realçar/unificar na nova visão de saúde.

Honestidade sobre limites (zero-dep): detectar índice ausente real, filesort ou nº de linhas retornadas exigiria rodar EXPLAIN ou capturar row counts — o Warden não faz isso hoje (precisaria conexão ao DB do child / mais captura). Fica de fora desta versão; podemos sinalizar como heurística futura.

Recomendação: 1 + 2 + 4 + 6 (maior valor/risco, baixo custo) e 3 + 5 como complementos baratos. Apresentados juntos numa visão "Saúde das queries" dentro de Database, cada achado linkando para os traces afetados (usa o drill-down).
